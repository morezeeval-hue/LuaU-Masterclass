Part 1

Luau wirklich lernen

Nicht auswendig lernen. Verstehen, warum Code so gebaut wird.

01

Variablen und Werte

Eine Variable speichert einen Wert unter einem Namen. In Luau benutzt du fast immer local, damit dein Code kontrolliert und vorhersehbar bleibt.

local coins = 50
local playerName = "Builder"
local isAlive = true

coins += 25
print(playerName, coins, isAlive)
Übung: Erstelle level, xp und requiredXp. Wenn xp groß genug ist, erhöhe das Level.
02

If, elseif, else

Kontrollfluss entscheidet, welcher Code läuft. In Spielen brauchst du das für Schaden, Preise, Unlocks, Cooldowns und Rechte.

local level = 12

if level >= 50 then
    print("Veteran")
elseif level >= 10 then
    print("Fortgeschritten")
else
    print("Anfänger")
end
Typischer Fehler: = weist zu, == vergleicht. Schreibe bei Bedingungen coins == 0, nicht coins = 0.
03

Funktionen

Funktionen machen aus wiederholtem Code ein Werkzeug. Eine gute Funktion hat einen klaren Namen, klare Parameter und möglichst eine Aufgabe.

local function addCoins(currentCoins: number, amount: number): number
    if amount <= 0 then
        return currentCoins
    end

    return currentCoins + amount
end

local newBalance = addCoins(100, 25)
Übung: Schreibe canBuy(price, coins), die true zurückgibt, wenn genug Coins vorhanden sind.
04

Tabellen

Tabellen sind Arrays, Dictionaries und Objekte zugleich. In Roblox-Systemen speicherst du damit Inventare, Profile, Konfigurationen und Caches.

local inventory = { "Sword", "Potion" }
table.insert(inventory, "Key")

local itemConfig = {
    Sword = { damage = 15, price = 100 },
    Potion = { heal = 25, price = 30 },
}

print(itemConfig.Sword.damage)
Merke: Arrays nutzt du für Listen. Dictionaries nutzt du, wenn du per Schlüssel schnell etwas finden willst.
05

Schleifen

Schleifen wiederholen Arbeit. Nutze sie für Inventarlisten, Spielerlisten, Spawnpunkte oder Cleanup.

local rewards = { 10, 20, 50 }

for index, reward in rewards do
    print(index, reward)
end

for _, player in game:GetService("Players"):GetPlayers() do
    print(player.Name)
end
Performance: Keine endlosen while true do-Schleifen ohne sinnvolles task.wait().
06

Typed Luau

Typen helfen dir, Fehler zu finden, bevor Spieler sie finden. Starte neue Module mit --!strict.

--!strict

type Item = {
    id: string,
    name: string,
    stackable: boolean,
    maxStack: number?,
}

local function describe(item: Item): string
    return `{item.name} ({item.id})`
end
Übung: Erstelle einen Typ Quest mit id, title, goal und rewardCoins.

Part 2

Roblox Studio verstehen

Roblox ist nicht nur Luau. Du codest gegen eine Engine mit Objekten, Services, Events und Replikation.

Wo welcher Code hingehört

Script läuft auf dem Server. Nutze es für Regeln, Schaden, Saves und wichtige Entscheidungen.

LocalScript läuft auf dem Client. Nutze es für UI, Kamera, Input, lokale Effekte.

ModuleScript gibt Werte oder Funktionen zurück. Nutze es, um Code zu teilen und Systeme sauber zu trennen.

-- ReplicatedStorage/Shared/MathUtil
local MathUtil = {}

function MathUtil.clampInt(value: number, min: number, max: number): number
    return math.clamp(math.floor(value), min, max)
end

return MathUtil

Server/Client mental model

Der Client kann manipuliert werden. Deshalb gilt: Der Client darf fragen, der Server prüft und entscheidet.

-- ServerScriptService/Shop.server.lua
local ReplicatedStorage = game:GetService("ReplicatedStorage")
local BuyItem = ReplicatedStorage.Remotes.BuyItem

local PRICES = { Sword = 100, Potion = 30 }
local cooldown: {[Player]: number} = {}

BuyItem.OnServerEvent:Connect(function(player, itemId: string)
    if typeof(itemId) ~= "string" then return end
    if not PRICES[itemId] then return end

    local now = os.clock()
    if cooldown[player] and now - cooldown[player] < 0.5 then return end
    cooldown[player] = now

    -- Hier coins serverseitig prüfen, abziehen, Item geben.
end)

Part 3

Systeme bauen

Hier lernst du nicht einzelne Befehle, sondern echte Bausteine für Spiele.

InventoryService

Ziel: Spieler bekommen Items, der Server speichert sie, UI zeigt sie an. Die Datenstruktur ist ein Dictionary: ItemId → Anzahl.

--!strict
local InventoryService = {}

type Inventory = {[string]: number}
local inventories: {[Player]: Inventory} = {}

function InventoryService.create(player: Player)
    inventories[player] = {}
end

function InventoryService.addItem(player: Player, itemId: string, amount: number)
    if amount <= 0 then return end
    local inventory = inventories[player]
    if not inventory then return end
    inventory[itemId] = (inventory[itemId] or 0) + amount
end

function InventoryService.get(player: Player): Inventory?
    return inventories[player]
end

function InventoryService.remove(player: Player)
    inventories[player] = nil
end

return InventoryService
Aufgabe: Ergänze removeItem. Sie soll niemals unter 0 gehen und true/false zurückgeben.

DataStore-Grundmuster

DataStores können fehlschlagen. Deshalb nutzt du pcall, Defaults, Autosave und speicherst nur serverseitig geprüfte Daten.

local DataStoreService = game:GetService("DataStoreService")
local Players = game:GetService("Players")
local store = DataStoreService:GetDataStore("profiles_v1")

local DEFAULT = { coins = 0, level = 1, inventory = {} }
local profiles = {}

local function loadProfile(player)
    local success, data = pcall(function()
        return store:GetAsync(`player_{player.UserId}`)
    end)

    profiles[player] = if success and typeof(data) == "table" then data else table.clone(DEFAULT)
end

local function saveProfile(player)
    local profile = profiles[player]
    if not profile then return end

    pcall(function()
        store:SetAsync(`player_{player.UserId}`, profile)
    end)
end

Players.PlayerAdded:Connect(loadProfile)
Players.PlayerRemoving:Connect(function(player)
    saveProfile(player)
    profiles[player] = nil
end)
Wichtig: Für große Spiele brauchst du zusätzlich Session-Locking, Retry-Strategien und UpdateAsync statt blindem SetAsync.

UI mit LocalScript

UI ist Client-Arbeit. Der Client rendert, der Server hält die Wahrheit. Sende nur Aktionen, nicht komplette neue Werte.

-- LocalScript in einem TextButton
local ReplicatedStorage = game:GetService("ReplicatedStorage")
local BuyItem = ReplicatedStorage.Remotes.BuyItem

local button = script.Parent
local itemId = "Sword"

button.Activated:Connect(function()
    button.Active = false
    BuyItem:FireServer(itemId)
    task.delay(0.5, function()
        button.Active = true
    end)
end)

Components mit CollectionService

Tags machen Objekte datengetrieben. Jedes Objekt mit Tag DamageZone bekommt automatisch Verhalten.

local CollectionService = game:GetService("CollectionService")

local function setupDamageZone(part: BasePart)
    part.Touched:Connect(function(hit)
        local character = hit.Parent
        local humanoid = character and character:FindFirstChildOfClass("Humanoid")
        if humanoid then
            humanoid:TakeDamage(10)
        end
    end)
end

for _, zone in CollectionService:GetTagged("DamageZone") do
    if zone:IsA("BasePart") then setupDamageZone(zone) end
end

CollectionService:GetInstanceAddedSignal("DamageZone"):Connect(function(zone)
    if zone:IsA("BasePart") then setupDamageZone(zone) end
end)

Part 5

Debugging, Qualität, Performance

Debugging-Checkliste

  • Output lesen. Die erste Fehlermeldung ist meist die wichtigste.
  • Mit print() Werte sichtbar machen, aber später entfernen.
  • Prüfen, ob Code auf Server oder Client läuft.
  • Mit typeof() Remote-Daten validieren.
  • Nil-Fehler durch frühe Returns abfangen.

Performance-Regeln

  • Keine unnötigen Loops pro Frame.
  • Events statt permanentes Polling verwenden.
  • Instanzen cachen, wenn du sie oft brauchst.
  • Remotes sparsam senden, Payload klein halten.
  • Server rechnet Regeln, Client macht Darstellung.
